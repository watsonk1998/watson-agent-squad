# Publishing Boundary

## Public

Safe to publish:
- methods
- templates
- sanitized skills
- generic scripts
- neutral examples
- architecture explanations

## Private

Keep private:
- real journals and reviews
- real Engram memories
- real MCP configuration
- real machine paths
- personal or employer/client-specific metadata
- screenshots with identifiable content
- git history that contains sensitive context

## Rule of thumb

Publish the **framework**, not the **life data**.

## Recommended repo split

- public repo: framework + templates + sanitized examples
- private vault / private repo: actual content and operational configs
